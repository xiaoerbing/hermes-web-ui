# 智能作业批改系统 — 落地方案

## 核心策略：Dify 验证 → 独立开发 → 渐进嵌入

不走"闷头全量自研"，也不走"一直挂在 Dify 上"。先用 Dify 快速验证流程，再逐步将核心模块独立开发，最终嵌入完整的项目软件中。

**模型选型**：使用 **Qwen3-VL** 系列多模态模型（阿里云百炼平台），一次调用同时完成"姓名提取 + 题目识别 + 批改归因"，无需额外商用 OCR API。

| 任务 | 推荐模型 | 单张成本 |
|---|---|---|
| 批量提取学生姓名 | Qwen3-VL-Flash | ~0.0003元 |
| 全页作业批改 | Qwen3-VL-Plus | ~0.003元 |
| 高难度推理题 | Qwen3-VL-235B-A22B | ~0.005元 |

---

## 阶段一：Dify 编排 + API 接入（快速验证）

**目标**：跑通全流程，让老师真实试用，验证效果。

- 用 Dify 拖拽搭建批改工作流：图片上传 → 多模态识别 → 批改 → 归因 → 报告
- **批量照片预处理**：老师将所有照片放进一个文件夹，运行 Python 脚本自动完成：
  1. 调用 Qwen3-VL-Flash 提取每张照片上的学生姓名（~0.0003元/张）
  2. 按学生分组，自动调用 Dify API 逐一批改
  3. 汇总导出全班批改报告（Excel）
- **前端直接调 Dify API**，零后端开发即可嵌入项目
- 评分标准、模型替换、题型分支调整全在 Dify 界面完成，不碰代码
- 收集真实反馈，迭代 prompt 和流程设计

> 这个阶段的工作流就是后续独立开发的 **技术规格说明书**。
> 详细 Dify 操作步骤见：[[Dify云服务操作指南-作业批改工作流]]

---

## 阶段二：核心模块独立开发（逐步替换 Dify 节点）

验证充分后，将性能敏感或需要深度定制的模块独立开发，以 API 形式嵌入 Dify 工作流：

| 独立开发模块 | 替换目标 | 优先级 |
|---|---|---|
| 图片预处理（OpenCV + YOLO 版面检测） | Dify 图像节点 | 🔴 高 |
| OCR 识别服务（PaddleOCR 本地部署） | Dify 识别节点 | 🔴 高 |
| 数据持久化（PostgreSQL + Neo4j 知识图谱） | Dify 数据节点 | 🔴 高 |
| 客观题规则引擎 | Dify 规则比对 | 🟡 中 |
| 学情统计分析 API | Dify 统计节点 | 🟢 低 |

Dify 工作流演变为：调用自研 API 完成预处理/OCR → Dify 做路由和大模型批改 → 调用自研 API 入库。

---

## 阶段三：完整独立应用（Dify 可选保留）

当核心模块全部独立后，架构升级为完整的自研系统：

- **前端**：小程序 / Web 管理端 / H5
- **后端**：自研批改引擎（预处理 → OCR → 题型路由 → 批改 → 归因统计）
- **存储**：PostgreSQL + Neo4j 知识图谱
- **大模型**：自研服务或仍走 Dify API

**Dify 可选保留为"评分标准管理后台"** — 教研老师可在界面上自行调整主观题评分 prompt 和标准，主流程通过 API 调用 Dify。

## 落地节奏

```text
第1-2周        第3-6周           第7-12周           第13周+
┌──────┐     ┌──────────┐      ┌────────────┐      ┌────────┐
│ Dify │ →  │ Dify +   │ →   │ 独立模块     │ →   │ 完成   │
│快速  │     │ 自研辅助  │      │ 逐步替换     │      │ 独立   │
│验证  │     │ 节点      │      │ Dify节点     │      │ 系统   │
└──────┘     └──────────┘      └────────────┘      └────────┘
 MVP版本      结构化优化版        深度定制版           生产就绪版
```

---

## 关键原则

- **不要一开始就追求完全独立开发** — Dify 先验证"这条路走得通"
- **Dify 里调好的 prompt 和流程 = 后续开发的技术规格书**
- **大模型调用可长期保留在 Dify** — 评分 prompt 需要老师频繁调整，可视化界面比改代码友好
